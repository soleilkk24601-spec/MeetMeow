# MeetMeow 开发者手册

## 项目概述

**MeetMeow（转角遇见咪）** 是一个基于 AI 视觉识别的城市猫咪互动记录微信小程序。

**核心价值（个体）**: 拍摄猫咪 → AI识别分析 → 生成故事卡片 → 收藏图鉴
**核心价值（组织）**: 浏览内容广场 → 查看LBS信息 → 地图 → 救助

---

## 技术架构
- 客户端：微信小程序原生栈（WXML/WXSS/JS），含拍照/上传、结果卡片、内容广场、用户中心、聊天悬浮球。
- 存储与媒体：阿里云 OSS 持久化图片与长图；小程序本地缓存保存用户档案、草稿、聊天历史。微信云开发 cloud1存储用户与猫咪信息
- AI 能力：Dify Workflow（图片理解、共创故事）与 Chatflow（聊天流式）统一封装在 `utils/api.js`、`utils/chat.js`。
- 服务端：Node.js 签名服务为 OSS PUT 生成签名；云开发数据库（`users`、`cats_data`、`share_posts` 等）直连读写。
- 地图与定位：腾讯地图 WebService 逆地理（合法域名 `https://apis.map.qq.com`），小程序 `<map>` 组件呈现示意点位。

## 全局智能问答助手

### 场景与目标

- 任何页面都能通过悬浮球唤起 Chat 页面，快速咨询猫咪护理、救助、养护等问题。
- UI 采用温暖黄色系，包含模型头像占位图、左右气泡、流式加载骨架，契合现有视觉调性。
- 接入 Dify Chatflow，首期只发送文本 query，预留系统提示、会话上下文以及多模态扩展能力。

### 技术要点

| 模块 | 说明 |
|------|------|
| `components/floatingBall` | fixed + movable 的悬浮球组件，支持拖拽、位置记忆、浸入式蒙层屏蔽。|
| `pages/chat` | IM 布局、滚动容器、输入框、发送按钮、失败重试、下拉加载历史占位。|
| `utils/chat.js` | `sendChatMessageStream()` 封装 Dify Chatflow 的 streaming 能力，提供 `onChunk`/`onComplete`/`onError` 回调。|
| 历史存储 | 先写本地缓存 `qa_sessions`，再择机同步 `assistant_sessions`/`assistant_messages` 云集合。|

### 智能助手核心流程

```
悬浮球点击 → 进入 Chat 页面 → 获取/创建 conversationId
→ 用户输入 query → sendChatMessageStream (response_mode=streaming)
→ SSE 分片解析 + 渲染进度 → 消息完成
→ 写入本地历史（可触发云端同步）
```

### 未来扩展

1. 会话列表与命名：允许从聊天页顶部切换历史记录。
2. 系统提示拼装：根据入口页面追加背景信息（如图鉴详情里的猫咪摘要）。
3. 富媒体：上传图片/音频后附带 URL 向 Chatflow 发送多模态内容。
4. 多模型：为进阶用户提供大模型选择与配额策略。

## 数据模型

### `user` 表

存储小程序自动登录（基于 open_id）的用户资料，每次登录后同步以下字段：

```json
{
  "_id": "openid",               // 与微信 open_id 对应
  "nickname": "MeetMeower",
  "avatar_url": "https://example.com/avatar.jpg",    //用户头像
  "role": "user",                // user / ngo / admin
  "is_lbs_enabled": false,        // 是否允许记录坐标
  "verify_status": "none",       // none/pending/verified/rejected
  "created_at": "2023-10-27T10:00:00.000Z",
  "updated_at": "2023-10-27T11:00:00.000Z"
}
```

> 说明：默认 `role=user`、`location_permisson=false`、`verify_status=none`。认证入口将 `verify_status` 更新为 `pending`，后台审核通过后写入 `role=ngo` + `verify_status=verified`。

### `cats_data` 表

- 保存每次 AI 识别后的图鉴记录（云数据库），包含以下字段：

```json
{
  "_id": "auto-id",
  "user_id": "openid",               // 关联 user 表
  "image_url": "https://oss/...",
  "basic_info": { "breed": "..." },
  "health_analysis": { "status": "..." },
  "emotion_analysis": { "intent_tag": "..." },
  "environment": "...",
  "warm_summary": "...",
  "raw_text": "Dify 原始文本",
  "structured_output": {...},
  "share_story": "AI 共创故事，来自 story 字段",
  "meet_time": "2025-12-04T07:30:00.000Z",
  "meet_location": "",  //默认为空
  "source": "dify"
}
```

**运行机制**

1. 用户进入“我的”页：

2. 切换 LBS、同步头像、提交认证都调用 `updateUserRecord` 写回 `users`。
3. 首页调用 `addToCloudCatalog` 时，通过 `getCurrentUserId()` 写入 `user_id`；图鉴页通过相同方法过滤 `cats_data`。

## 页面功能

### LBS 功能

**角色与权限**

- `user`：默认角色，可自主开启 `location_permission`，上传时可选择记录 GPS；在广场/内容流仅看到模糊位置信息或空值。
- `ngo`：认证动保组织账号，可在广场看到精确经纬度并打开地图导航，便于救助/TNR。

**认证流程**

1. 用户中心新增“动保组织认证”入口。
2. 表单字段：机构名、联系人、证件照/文件、说明。
3. 提交后状态标记为 `pending`；后台审核通过则更新账号 `role=ngo`、`location_permission=true`。

**设置与权限**

- 用户中心新增 `location_permission` 开关（默认 `false`）。
- 当用户开启时需调用 `wx.authorize({ scope: 'scope.userLocation' })`。
- 开关状态写入用户资料表，与本地缓存同步。

**拍照上传流程**

1. 触发上传→判断 `location_permission`。
2. 若开启且授权成功：调用 `wx.getLocation`，将 `latitude/longitude` 写入上传 payload。
3. 记录结构：`encounter_logs` 表新增字段 `location_enabled`, `latitude`, `longitude`, `geo_hash`（便于模糊查询）。

**广场渲染（安全策略）**

| 请求者角色 | 后端返回 | 前端展示 |
|-------------|-----------|-----------|
| `user` | `latitude`,`longitude` 置空或裁剪成城市-区县字段 | 最多展示模糊标签 `📍 北京市·朝阳区` |
| `ngo` | 原样返回精确坐标 | 展示 `📍` 图标，点击进入地图弹窗或导航 |

若帖子的定位被裁剪/空值，前端需隐藏地图按钮，只展示“无定位”文案。

### 1. 首页 (index) - 核心功能页

**功能模块**:

| 功能 | 方法 | 说明 |
|------|------|------|
| 拍照 | `takePhoto()` | 调用摄像头拍摄 |
| 选图 | `chooseFromAlbum()` | 从相册选择 |
| 获取签名 | `requestOssSignature(filename, contentType)` | 请求 Node.js 服务生成 Authorization Header |
| 上传 OSS | `uploadToOss(filePath)` | 用 PUT + Authorization Header 上传，得到公网 URL |
| 调用 AI | `callDifyApi(imgUrl)` | 触发 `callImageWorkflow`（`utils/api.js`），传入图片 URL + 文本描述 |
| 解析结果 | `prepareAnalysisRecord()` | 解析 Dify JSON 字符串并填充 `catResult` |
| 互动工作流 | `handleUserInteraction()` + `callInteractionWorkflow()` | 文本/语音消息 -> 归一化上传 -> 调用 Dify 互动工作流，展示回复 |
| 收藏 | `collectCat()` | 写入本地存储（后续接云数据库） |
| 分享 | `shareCat()` | 调用微信分享能力 |
| 保存记录 | `saveToDatabase(data)` | 云数据库预留|


**结果卡片浮层内容**:
- 上半部分（约 80% 高度）：猫咪图片 + 基本信息 + Dify 结构化输出，放入 `scroll-view` 中滚动展示。
- 下半部分（固定底栏）：语音/文字输入栏 + 操作按钮。
  - 输入栏支持键盘输入，可选语音。。
  - “加入图鉴”“分享”两个按钮固定在底部。


### 2. 图鉴页 (collects)

- 读取云数据库 `cats_data` 并以 3 列自适应卡片展示，每张卡片显示封面与情绪标签。
- 点击卡片进入详情浮层，可查看基础信息、输入故事（键盘+语音）并调用 `callInteractionWorkflow` 获取 AI 共创回复。
- 详情浮层提供删除按钮，确认后会同步删除云端记录并更新本地列表。

### 3. 内容广场 (share)

#### 功能目标

| 模块 | 业务目的 |
|------|----------|
| 分享入口 | 将首页结果卡片、图鉴详情卡片转为帖子，附带封面长图与文字，沉淀到内容广场 | 
| 信息流展示 | 朋友圈式瀑布流，混排头像、昵称、文案、长图、点赞；普通用户隐去精确 LBS，组织（`role=ngo`/`admin`）展示精确定位 | 
| 互动 | 仅支持点赞（可重复取消），不做评论；作者可删除自己的帖子 | 

#### 分享入口设计

1. **首页结果卡片 > 分享按钮**
  - 点击后进入 `pages/share/editor`（待新增）
  - 默认携带：`catResult` 的长图（使用当前卡片渲染导出的图片，或临时生成海报 URL）、`shareStory`/`warm_summary` 摘要、`meet_location`、`meet_time`、`catname`
  - 用户可编辑：文本内容（最长 500 字）、是否公开定位（默认同用户 `is_lbs_enabled` 设置）、是否标记猫咪名
2. **图鉴详情卡片 > 分享按钮（删除按钮右侧）**
  - 逻辑同上，差异在于数据来源 `activeCat`
3. **编辑页保存逻辑**
  - 表单提交 → 写入 `share_posts` 集合
  - 成功后跳转内容广场列表页并 toast “已发布”，失败弹窗+保持草稿

#### 数据模型 (share_posts)

```json
{
  "_id": "auto-id",
  "author_id": "openid",
  "author_profile": { "nickname": "...", "avatar": "..." },
  "cat_ref_id": "cats_data._id",          // 非必填
  "media_url": "https://oss/...",         // 长图/卡片截图
  "content": "今天遇到...",               // 用户自定义文本
  "share_story": "AI 共创文案",            // 直接复用或为空
  "lbs": {
   "raw": { "lat": 30.1, "lng": 114.3 },
   "display": "武汉·武昌区"              // 预裁剪文本
  },
  "visibility": {
   "show_lbs_to_public": false            // 表单开关，默认读取用户 is_lbs_enabled
  },
  "likes": {
   "count": 0,
   "user_ids": []                         // 考虑到小程序读写限制，可另建 like 集合
  },
  "created_at": "ISO",
  "updated_at": "ISO",
  "status": "active"                      // active / deleted
}
```

> **用户体系关联**：`author_id` 使用 `getCurrentUserId()`；若用户未初始化账号则禁止进编辑页。`author_profile` 在发布时快照，避免后续更名导致老帖显示异常。

#### 信息流 UI/逻辑

1. **列表页**
  - WeUI 列表 + 卡片组合；单位模块结构：头像+昵称+时间 → 文本 → 图片 → LBS 文案 → 点赞/删除/分享操作区
  - 采用分页查询（`limit 10` + `skip`）或 `cloud.callFunction('shareFeed')` 返回 {data, hasMore}
  - LBS 文案策略：
    - 普通用户（role=user）→ 若 `visibility.show_lbs_to_public=false` 则隐藏；若 true 则展示模糊地点（`lbs.display`）
    - 组织用户（role=ngo/admin）→ 若 `lbs.raw` 存在则显示精确坐标按钮（`wx.openLocation`）；若缺失则降级为 display
2. **点赞**
  - 单独的 `share_likes` 集合或在文档内维护 `liked_user_ids`（预估数据量小时可直接更新数组）；逻辑：再次点击即取消
  - 前端使用本地状态 `post.isLiked` 与 `post.likeCount`，成功后同步云端
3. **删除**
  - 仅 `author_id === currentUserId` 的帖子展示“删除”按钮
  - 删除方式：软删除（`status=deleted`），feed 查询默认过滤；成功后在前端移除

#### 编辑页交互

| 字段 | 默认值 | 说明 |
|------|--------|------|
| 封面图 | 调用截图/海报生成 API，失败时回退到 `image_url` | 只读，点击可预览 |
| 文本输入 | `share_story` 或空 | 可编辑，500 字限制 |
| LBS 可见开关 | 继承 `user.is_lbs_enabled` | 切换后写入 `visibility.show_lbs_to_public` |
| 猫咪标签 | `catname` 或品种 | 勾选控件，允许用户隐藏 |

发布时需要再次调用 `getCurrentUserId()`，若无返回则提示“请先在‘我的’页初始化账号”。

| 功能 | 位置 | 描述 |

### LBS 实现与展示

- 定位采集与逆地理
  - 拍照页记录 `meet_location={ latitude, longitude }`，图鉴/分享草稿用 `locationRaw` 传递到编辑页。
  - 编辑页调用腾讯位置服务 `reverseGeocoder`（域名 `https://apis.map.qq.com` 需加入小程序 request 合法域名，使用 WebService Key）将经纬度解析为城市名，写入 `lbs.display`；原始坐标放 `lbs.raw`。
- Share 列表展示策略（权限不变）
  - `ngo/admin`：显示 `displayLbs`，若缺失回落到 `util.formatLocationTag(lbs.raw)`（可能是经纬度）；可点击查看示意地图。
  - 普通用户：仅在 `visibility.show_lbs_to_public=true` 时展示 `lbs.display`，不会显示精确经纬度。
- Mock-map 页面数据来源
  - 由 share 列表 `openMockMap` 通过页面参数传入 `lat/lng`（可选标题/地址/距离/时长）；[pages/share/mock-map/mock-map.js](pages/share/mock-map/mock-map.js) 在 `onLoad` 读取并生成 `markers`，使用小程序内置 `<map>` 渲染，没有额外外部 API 调用。

#### 与现有页面的集成点

1. **首页结果卡片**：`shareCat` 按钮新增“分享到内容广场”选项（ActionSheet 形式）。点击后跳转编辑页并带上 `scene=analysis`，附带 `catResult + shareStory + uploadedImgUrl + meet_location`
2. **图鉴详情卡片**：在“删除”右侧新增“分享”按钮，`scene=collects`，附带 `activeCat`
3. **TabBar**：`pages/share/share` 作为内容广场列表页入口。Tab 图标保持一致（可显示“内容广场”文案）
4. **用户中心**：
  - 展示“已发布 X 条内容”统计（查询 share_posts.author_id=currentUserId）
  - 管理入口：跳转到“我的分享”列表，可批量删除

#### 权限 & 风控

- 所有分享必须由登录用户发起，且需绑定 `user_profile`
- 暂不提供匿名模式

> 编码前请确认 OSS 海报生成方案（前端 2D canvas 或云函数截图）以及云数据库索引（`author_id`, `created_at`）。

### 4. 用户中心 (user)

- 入口展示：头像、昵称、角色、认证状态、创建时间。
- 关键逻辑：
  1. `bootstrapUserProfile()`：读取调试 openid 或本地缓存，尝试从 `users` 集合获取文档；若不存在则创建默认记录。
  2. `handleGetProfile()` → `updateUserRecord()`：同步微信头像/昵称。
  3. `toggleLbs()`：授权定位后写入 `is_lbs_enabled`。
  4. `submitVerifyApplication()`：表单校验→`verify_status=pending`。
  5. “调试账号切换”：设置 `debug_user_override_openid`，供图鉴/上传流程读取。
- 开发阶段默认启用调试 openid，真实登录逻辑待上线前接入云函数。

### 5. 全局智能助手 (chat)

| 功能 | 说明 |
|------|------|
| 悬浮球入口 | 所有主页面挂载 `<floating-ball>`，支持拖动并记忆位置，点击后 `wx.navigateTo('/pages/chat/chat')`。|
| 会话初始化 | `conversationId` 缓存在本地，若不存在则调用 `createConversationId()`（时间戳+随机串）。|
| 消息列表 | `scroll-view` 自底向上渲染，分为 `user`、`assistant`、`loading` 三种气泡，气泡宽度随内容自适应，头像使用 PNG 占位。|
| 输入与发送 | 多行输入框 + 发送按钮；禁用状态：空文本或当前存在流式请求。|
| 流式输出 | `utils/chat.js` 通过 `wx.request` + `enableChunked` 解析 SSE，实时更新正在生成的气泡，结束后写入历史数组。|
| 错误处理 | 流式中断/HTTP 错误时更新消息状态为 `failed`，提供“重新发送”按钮。|
| 历史扩展 | 预留顶部“会话菜单”按钮，未来接入列表/重命名；新增 `localHistoryStore`，统一封装读写接口。|

UI 设计原则：背景渐变（浅米黄→暖灰），消息气泡使用 #FDEFC2/#FFF7E3，输入区使用圆角卡片并保持与主场景一致的字体与 icon 集。

### Dify 图片理解工作流（图片）

- **封装**：`utils/api.js` 中维护 `IMAGE_DIFY_API` 与 `callImageWorkflow({ imageUrl, userId })`。
- **调用链**：`pages/index/index.js` 的 `callDifyImageApi()` 只负责触发 `callImageWorkflow` 并解析结果，不会在页面内暴露密钥。
- **请求参数**：`inputs.image_url` 直接传入 OSS 公网链接，`response_mode = 'blocking'`。
- **输出格式**：工作流返回text变量下的JSON 字符串。`callImageWorkflow` 内部会做 `status` 校验，页面端继续使用 `prepareAnalysisRecord` 落地结构化数据。
- **异常处理**：当 `status !== succeeded`、HTTP 非 200 或 `text` 为空时，`callImageWorkflow` 会 reject，对应页面捕获后友好提示（如「AI 正在处理中」「AI 无内容返回」）。

### Dify 用户故事互动工作流（文字/语音）

- **封装**：`utils/api.js` 维护 `INTERACTION_DIFY_API` 与 `callInteractionWorkflow({ imageUrl, text，audio })`，统一校验图片 URL 与文本/语音内容。
- **请求格式**：音频、文本、图片 URL 通过 `inputs.audio` / `inputs.text` / `inputs.image` 传给 workflow，至少需要文本或音频，图片 URL 必填。
- **图片来源**：
  - 首页结果卡片：优先使用最新一次上传成功的 `uploadedImgUrl`，若用户关闭卡片后重新打开，则回退到 `catResult.image_url`。
  - 图鉴详情浮层：使用 `activeCat.image_url`，若记录缺少远程地址则禁止发送共创请求。
- **前端入口**：`handleUserInteraction()`（首页）与 `handleSendStory()`（图鉴）都会先保证图片 URL 存在，再根据 `userPrompt` / `voiceTempPath` 或 `storyVoicePath` 判断文本/语音；语音分支通过 OSS 上传复用 PUT 流程拿到公网 URL。
- **响应解析**：工作流 `outputs.text` 返回 JSON 字符串，字段包含 `error`、`au_text`、`story`。前端会先解析 JSON；若存在 `error` 则弹窗提示并停止展示；若无错误，仅使用 `story` 文本渲染共创卡片。
- **UI 逻辑**：结果卡片和图鉴详情均提供“发送”按钮（`canSendInteraction` / `canSendStory` 控制高亮/禁用）。只有当输入框有字符或成功录到音后才允许发送；点击按钮会展示 Loading，并在成功后清空输入框与音频缓存、展示最新 `story`，并允许重复生成覆盖旧内容。
- **分享字段**：最新 `story` 文本会暂存为 `share_story`。在首页点击“加入图鉴”时会把 `share_story` 一并写入 `cats_data`；在图鉴详情中生成故事后，也会直接更新对应记录的 `share_story` 字段并同步到本地列表。
- **错误提示**：语音上传失败走 `wx.showToast('语音上传失败')`；互动工作流若返回非 200 或 `outputs.text` 空则提示“发送失败/AI 无返回”。

## 后端签名服务 (Node.js)

- 位置：`server/app.js`
- 技术：Express + 原生 `crypto`
- 职责：根据阿里云官方文档手工计算 PUT 请求的 `Authorization` Header，确保 `Date` 与 `Content-Type` 与前端上传保持一致，防止 403。

### 启动步骤

1. 进入 `server/` 目录执行 `npm install`
2. 复制 `.env.example` 为 `.env` 并填入真实 OSS 配置
3. 运行 `npm start`（默认端口 `3000`）

### API

- `GET /api/oss/sign-put`
  - Query: `filename`（必须）、`contentType`（可选，默认 `application/octet-stream`）
  - 行为：
    1. 生成 `gmtDate = new Date().toGMTString()`；
    2. 拼接 `CanonicalizedResource = /<bucket>/<filename>`；
    3. 按顺序组装 `StringToSign = "PUT\n\n${contentType}\n${gmtDate}\n${canonicalizedResource}"`；
    4. 使用 `HMAC-SHA1(AccessKeySecret, StringToSign)` 并做 Base64 得到签名；
    5. 返回 `Authorization: OSS <AccessKeyId>:<signature>` 以及 `uploadUrl`、`publicUrl`。
  - Response:

    ```json
    {
      "code": 200,
      "data": {
        "uploadUrl": "https://<bucket>.<region>.aliyuncs.com/images/xxx.jpg",
        "headers": {
          "Authorization": "OSS LTAI...:abc=",
          "Date": "Thu, 28 Nov 2025 08:12:00 GMT",
          "Content-Type": "image/png"
        },
        "publicUrl": "https://<bucket>.<region>.aliyuncs.com/images%2Fxxx.jpg",
        "objectKey": "images/xxx.jpg"
      }
    }
    ```

前端必须原样使用返回的 `uploadUrl` 与 `headers` 调用 `wx.request` (PUT)。任何改动（如重新生成 Date、修改 Content-Type）都会导致 `SignatureDoesNotMatch`。

### 前端 PUT 上传流程

1. 通过 `buildObjectKey(dir, filePath)` 生成唯一文件名，并推断 `contentType`；
2. 请求 `GET /api/oss/sign-put?filename=...&contentType=...`，拿到 `uploadUrl` 与 `Authorization/Date/Content-Type`；
3. 使用 `wx.getFileSystemManager().readFile` 读取原始二进制；
4. 调用 `wx.request`，`method: 'PUT'`，`url` 为 `uploadUrl`，`header` 完全使用后端返回的 `headers`，`data` 直接传入二进制缓冲；
5. 状态码 200/204 视为成功，再把 `publicUrl` 传给 Dify 或存库。

## OSS 上传调试与常见问题

| 错误 | 可能原因 | 解决方案 |
|------|----------|----------|
| `getaddrinfo ENOTFOUND <bucket>.aliyuncs.com` | OSS 域名不存在或写错、`OSS_REGION` 拼写错误 | 以 OSS 控制台“外网访问域名”为准，确认 `OSS_REGION` 包含 `oss-` 前缀并重启签名服务 |
| `statusCode: 403` （XML 返回 `SignatureDoesNotMatch` 等） | policy/签名与上传字段不匹配、`key` 不在 `dir` 前缀内、超出大小限制 | 打印 `formData` 检查 `key`、`policy`、`OSSAccessKeyId` 与后端返回值一致；确保文件小于 `OSS_MAX_SIZE_MB` |
| `AccessDenied`/`InvalidAccessKeyId` | AccessKey 无权限或已失效 | 使用主账号或具备 `oss:PutObject` 权限的 RAM 子账号，更新 `.env` 并重启 |
| 上传成功但前端不可访问 | OSS 未设置公网读权限 | 在 OSS 控制台将 bucket ACL 设为公共读，或按需开启签名下载 |

调试建议：
1. `npm start` 后用 `curl` 或 Postman 模拟表单上传，验证签名有效；
2. 小程序端 `console.log(config.host, formData.key)`，确保与后端响应一致；
3. 开发阶段在微信工具里勾选「不校验合法域名」，上线前需在后台配置 request/uploadFile 白名单。
